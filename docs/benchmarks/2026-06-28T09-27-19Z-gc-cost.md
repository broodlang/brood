# GC cost study — copying collector on/off (for ADR-114)

- **UTC:** 2026-06-28T09-27-19Z
- **commit:** 6f8850d (tree dirty — instrumentation in progress)
- **cpu:** 11th Gen Intel(R) Core(TM) i5-11500H @ 2.90GHz (12 cores)
- **kernel:** Linux 7.0.0-22-generic
- **rustc:** rustc 1.94.0 (4a4ef493e 2026-03-02)
- **build:** `cargo build --release --features jit --bin brood` (NO debug-assertions; `:debug-build false` confirmed via gc-stats)

## Method

A/B each workload **GC-on** (default) vs **GC suppressed** (`BROOD_GC_FLOOR=500M` → adaptive
threshold pinned at 524,288,000 objects → 0 collections for these bounded workloads). The wall-time
delta is the **current copying collector's** cost (collection work ± its compaction/locality effect
on the mutator). Min of 6 runs each; startup baseline (`(println 1)`) = 0.02s, negligible.
Survivor rate = `copied / (copied + reclaimed)` from `(gc-stats)`.

## Results

| workload | survivor | collections | copied | reclaimed | GC-on | GC-off | Δ |
|----------|----------|-------------|--------|-----------|-------|--------|---|
| fib 32 (compute-bound)        | —    | 0   | 0          | 0          | 0.08s | 0.08s | 0% |
| listsum (build+fold 20k ×1500)| 14%  | 474 | 4,857,132  | 29,697,750 | 0.93s | 1.57s | **−40% (copying faster)** |
| bintree (depth-16 ×300)       | 60%  | 451 | 58,994,935 | 39,190,814 | 8.64s | 5.42s | **+37% (copying is the cost)** |

(Small/unscaled variants, for reference — survivor rates consistent: nqueens 0.24% / 1 collection,
listsum-small 14%, bintree-small 41%.)

## Reading

1. **Compaction is sometimes a net win.** listsum: GC-on beats GC-off by 40% because compaction
   keeps the working set cache-hot; the un-collected heap grows (~1.4 GB) and thrashes. A
   non-moving (mark-sweep) heap forfeits this — it stays bounded (slot reuse) so it won't thrash
   like GC-off, but it won't compact either, and it pays free-list vs bump-pointer allocation
   (~1.5–3× per alloc, literature) on *every* workload.
2. **Copying's cost is real only on high-survivor allocation-pathological workloads.** bintree
   (60% survivor — the GC-stress benchmark by design) is where copying hurts (+37%); mark-sweep
   would likely win there (marks the live set but never copies it). Minority case.
3. **Verdict:** throughput is roughly a wash → does not decide moving-vs-mark-sweep. The decision is
   stale-handle-bug-class elimination + kernel simplicity vs. rewrite cost + losing
   bump-allocation/compaction. See ADR-114 (recommendation: keep moving GC, add JIT stack maps).

## Workload sources

`bintree` (cons-tree build/check, intermediate-lived), `listsum` (build N-element list then
fold, medium survivor), `nqueens` (backtracking, near-zero survivor), `fib` (compute-bound, no
allocation pressure). Kept in the session scratchpad; reproduce by timing each `--release` run
with and without `BROOD_GC_FLOOR=500M`.
